1. プロジェクト概要
================================================================================

■ アプリケーション名
  VALORANT Custom Team Builder

■ 目的
  VALORANTのカスタムマッチにおいて、プレイヤーのランク情報に基づき
  公平なチーム分けを自動で行うWebアプリケーション。
  ホストが部屋を作成し、URLを共有するだけで参加可能。

■ 対象ユーザー
  VALORANTのカスタムマッチを遊ぶプレイヤーグループ（2〜10人想定）

■ 対応言語
  日本語

================================================================================

3. システムアーキテクチャ
================================================================================

  [ブラウザ (React)]
       │
       ├─ GET/POST/PATCH/DELETE → Next.js API Routes (/api/*)
       │                              │
       │                              ├─ Supabase (PostgreSQL)
       │                              └─ Henrik-3 Valorant API
       │
       └─ Supabase Realtime (リアルタイム同期)

■ デュアルモード設計
  - Supabase モード: 環境変数に SUPABASE_URL / SUPABASE_ANON_KEY が設定済みの場合
  - デモモード: 未設定時。インメモリ Map でデータ保持（globalThis で HMR 対応）

4. データベース設計
================================================================================

4.1 rooms テーブル
--------------------------------------------------------------------------------
  カラム名       | 型                                      | 説明
  --------------|----------------------------------------|-------------------
  id            | UUID (PK, default: gen_random_uuid())   | ルーム一意識別子
  created_at    | TIMESTAMPTZ (default: now())            | 作成日時
  status        | TEXT ('waiting'|'calculating'|'finished')| ルーム状態
  rank_mode     | TEXT ('current'|'peak', default:'current')| ランク計算モード

  インデックス: idx_rooms_created_at (created_at)

4.2 players テーブル
--------------------------------------------------------------------------------
  カラム名         | 型                                     | 説明
  ----------------|----------------------------------------|-------------------
  id              | UUID (PK, default: gen_random_uuid())   | プレイヤー識別子
  room_id         | UUID (FK → rooms.id, CASCADE DELETE)    | 所属ルーム
  riot_id         | TEXT (NOT NULL)                         | Riot ID (例: Player#1234)
  display_name    | TEXT                                    | 表示名
  rank            | TEXT                                    | 現在ランク
  rank_value      | INTEGER (default: 0)                   | 現在ランク数値 (1〜25)
  peak_rank       | TEXT                                    | 最高ランク
  peak_rank_value | INTEGER (default: 0)                   | 最高ランク数値 (1〜25)
  team            | TEXT ('A'|'B'|NULL)                     | チーム割り当て
  created_at      | TIMESTAMPTZ (default: now())            | 作成日時

  インデックス: idx_players_room_id (room_id)

4.3 自動クリーンアップ
--------------------------------------------------------------------------------
  - ルーム作成時に cleanupExpiredRooms() を実行
  - 作成から24時間以上経過したルームを自動削除
  - CASCADE DELETE によりプレイヤーも連動削除

5. 機能要件
================================================================================

5.1 ルーム管理
--------------------------------------------------------------------------------

  【ルーム作成】
  - トップページの「部屋を作る」ボタンで UUID v4 ベースの新規ルーム生成
  - 初期ステータス: waiting
  - 作成後 /room/[roomId] へ自動遷移

  【ルーム参加】
  - URLまたはルームコード入力で参加
  - ルームコード入力時、URL からルームIDを正規表現で自動抽出

  【ルームURL共有】
  - クリップボードAPIでルームURLをワンクリックコピー

5.2 プレイヤー管理
--------------------------------------------------------------------------------

  【プレイヤー追加 — API取得モード（デフォルト）】
  - Riot ID (Name#Tag) 入力 → Henrik-3 API から自動取得
    - 表示名
    - 現在ランク + 数値化
    - 最高ランク + 数値化
  - API未構成時はデモモードで動作（ランダムランク生成）

  【プレイヤー追加 — 手動入力モード】
  - プレイヤー名、現在ランク、最高ランクを手動選択・入力

  【プレイヤー削除】
  - 個別のプレイヤーをルームから削除可能

  【重複防止】
  - 同一 Riot ID の重複登録を防止

5.3 ランクシステム
--------------------------------------------------------------------------------

  ランク体系（26段階）:

  ティア        | サブランク  | 数値
  -------------|-----------|------
  Iron         | 1, 2, 3   | 1〜3
  Bronze       | 1, 2, 3   | 4〜6
  Silver       | 1, 2, 3   | 7〜9
  Gold         | 1, 2, 3   | 10〜12
  Platinum     | 1, 2, 3   | 13〜15
  Diamond      | 1, 2, 3   | 16〜18
  Ascendant    | 1, 2, 3   | 19〜21
  Immortal     | 1, 2, 3   | 22〜24
  Radiant      | —         | 25

  デフォルト値: Silver 2 (8)

  ランクモード選択:
  - current: 現在ランクを使用して計算
  - peak: 最高ランクを使用して計算

5.4 チーム振り分け（4モード）
--------------------------------------------------------------------------------

  【⚡ 自動バランス】
  - 動的計画法（部分和問題）ベースの最適化アルゴリズム
  - 処理フロー:
    1. プレイヤーリストをシャッフル（ランダム性確保）
    2. ランク合計の半分をターゲットとして設定
    3. DPテーブルで最適なチーム分けを探索
    4. チームA/Bのランク合計差が最小になる組み合わせを選択
  - 計算量: O(n × targetTeamSize × totalSum)
  - ランクモード (current/peak) 選択対応

  【🎲 完全ランダム】
  - Fisher-Yatesシャッフルで完全ランダム分け
  - ランク考慮なし

  【👑 ドラフトモード】
  - Phase 1: 各チームのリーダー選択
  - Phase 2: リーダーが交互にプレイヤーをピック
  - ピック被り対応: 同一プレイヤーを選択した場合、ランダムで勝者決定
    → 敗者は補償ピック獲得
  - 残り1人の場合: 人数の少ないチームへ自動割り当て

  【✋ 手動振り分け】
  - ドラッグ＆ドロップでプレイヤーをチーム間移動
  - リアルタイムでランク合計・差分を表示

  【チーム確定後の操作】
  - 🔄 再生成: 同モードで再実行
  - ドラッグ＆ドロップ: チーム間プレイヤー入れ替え
  - リセット: プレイヤー追加画面に戻る

  【チーム振り分け結果】
  - TeamResult 構造:
    - teamA / teamB: プレイヤー配列
    - teamATotal / teamBTotal: ランク合計値
    - difference: 両チームのランク差（絶対値）

5.5 エージェントBANシステム
--------------------------------------------------------------------------------

  【チームBANモード】
  - 各チームが交互に1キャラずつBAN（合計2キャラ）

  【個人投票BANモード】
  - 全プレイヤーが各2キャラを投票
  - 得票数上位2キャラをBAN

5.6 エージェントランダムピック
--------------------------------------------------------------------------------

  【Tier考慮版】
  - マップ別エージェントTier（S〜D、5段階）を考慮
  - 強いチーム → 低Tierエージェント優先
  - 弱いチーム → 高Tierエージェント優先
  - ロール別に最低2人確保

  【シンプル版】
  - Tier無視の純粋ランダム選択

  【共通仕様】
  - BANエージェントは除外
  - 各プレイヤーに異なるエージェントを割り当て
  - アニメーション表示（10フレーム × 100ms）
  - 個別エージェント再抽選（リロール）対応

5.7 エージェントTier管理
--------------------------------------------------------------------------------

  - マップ別にエージェントの強さをS〜Dでランク付け
  - S=5点, A=4点, B=3点(デフォルト), C=2点, D=1点
  - localStorageに保存・読込
  - Tier順にエージェントをソート・グループ化可能

5.8 マップ選択
--------------------------------------------------------------------------------

  対応マップ（11種）:
    Abyss, Ascent, Bind, Breeze, Fracture,
    Haven, Icebox, Lotus, Pearl, Split, Sunset

  選択方式:
  - 🎲 ランダム: ランダムでマップを1つ選択
  - 🗳️ 投票: 全プレイヤーがマップに投票 → 最多票を選択（同票時ランダム）

6. APIエンドポイント
================================================================================

6.1 POST /api/rooms
--------------------------------------------------------------------------------
  機能: ルーム新規作成
  レスポンス: { room: { id, created_at, status, rank_mode } }

6.2 GET /api/rooms/[roomId]
--------------------------------------------------------------------------------
  機能: ルーム情報 + プレイヤー一覧取得
  レスポンス: { room: Room, players: Player[] }

6.3 POST /api/rooms/[roomId]
--------------------------------------------------------------------------------
  機能: プレイヤーをルームに追加
  リクエスト: { riot_id, display_name, rank, rank_value, peak_rank, peak_rank_value }
  レスポンス: { player: Player }

6.4 PATCH /api/rooms/[roomId]
--------------------------------------------------------------------------------
  機能: ルーム/プレイヤー情報更新（action で分岐）

  action: 'updateTeam'
    → プレイヤーのチーム割り当て変更
    Body: { action, playerId, team }

  action: 'updateStatus'
    → ルームステータス変更
    Body: { action, status }

  action: 'updateRankMode'
    → ランクモード変更
    Body: { action, rankMode }

  action: 'resetTeams'
    → 全プレイヤーのチーム割り当てリセット
    Body: { action }

6.5 DELETE /api/rooms/[roomId]?playerId={id}
--------------------------------------------------------------------------------
  機能: プレイヤーをルームから削除
  レスポンス: { success: true }

6.6 GET /api/valorant?name={name}&tag={tag}
--------------------------------------------------------------------------------
  機能: Valorantプレイヤー情報取得（Henrik-3 API経由）
  レスポンス: { name, tag, rank, rankValue, peakRank, peakRankValue, displayName }
  フォールバック: API未設定/エラー時はデモモード（ランダムランク生成）

11. 非機能要件
================================================================================

  【パフォーマンス】
  - チーム振り分けアルゴリズムは10人規模で即時完了すること
  - API取得は非同期処理でUIをブロックしないこと

  【可用性】
  - デモモード搭載により、外部API/DB未接続でも動作可能
  - API失敗時はデモモードへ自動フォールバック

  【データ管理】
  - 24時間経過したルームの自動削除
  - ルーム削除時のプレイヤーデータ連動削除（CASCADE）

  【セキュリティ】
  - Valorant APIキーはサーバーサイドのみで使用（環境変数）
  - Row Level Security (RLS) はSupabase側で無効化（カスタム認証不要のため）

  【拡張性】
  - Supabase / デモモードの切り替えが環境変数のみで可能
  - エージェント・マップデータは定数ファイルで管理（追加容易）
